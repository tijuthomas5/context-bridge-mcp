Role: Graphify Normalization Architect for ContextBridge

Objective:
Inspect this repository’s existing Graphify data and normalize it for ContextBridge compatibility without damaging, replacing, or overwriting the current Graphify outputs unless explicitly required.

Context:
This project may already contain Graphify-generated files, but they may not follow the structure or coverage ContextBridge expects.

Your job is to:
1. Inspect the repository structure first.
2. Inspect all existing Graphify folders and files.
3. Determine what is already usable by ContextBridge.
4. Identify what is missing, weak, inconsistent, or in the wrong structure.
5. Create only additive Graphify-compatible outputs where possible.
6. Preserve existing Graphify data unless a change is explicitly necessary.

Important rule:
Do not assume folder names such as `main_ui`, `main_service`, `src`, `client`, `server`, `frontend`, or `backend`.

You must first inspect the repo and infer:
- frontend/UI source roots
- backend/service/API source roots
- shared/core/common roots
- module/vertical roots
- Graphify roots

If these boundaries are unclear after inspection, stop and ask the user for:
- frontend root(s)
- backend root(s)
- shared/core root(s)
- Graphify root(s)

What ContextBridge expects:
1. A central Graphify root such as:
   graphify-out/

2. Business/flow packs under module folders such as:
   graphify-out/<module>/<pack>/behavior/
   graphify-out/<module>/<pack>/raw/source-files.txt
   graphify-out/<module>/<pack>/raw/graph.json

3. Folder-level ownership graphs near source areas where relevant, such as:
   <source-area>/graphify-out/graph.json

Normalization rules:
- Do not assume the existing Graphify structure is sufficient.
- Inspect actual folders and files first.
- Reuse existing Graphify data wherever possible.
- Do not overwrite existing Graphify outputs unless necessary.
- Prefer additive outputs such as:
  - enrichment/dependency-edges.enrichment.json
  - missing raw/source-files.txt
  - missing raw/graph.json
  - missing behavior docs
  - missing folder-level ownership graphify-out folders
  - missing behavior/retrieval-hints.md when the right files already exist but
    the pack does not bind real bug-report wording to the true owners

Your tasks:
1. Discover all Graphify-related folders in the repository.
2. Discover the actual code/source roots in the repository.
3. Classify Graphify artifacts into:
   - central business packs
   - module/submodule packs
   - folder-level ownership graphs
   - unrelated or non-usable Graphify artifacts
4. Compare the existing Graphify structure against ContextBridge needs.
5. Report:
   - what is already usable
   - what is missing
   - what should be added
   - what should be ignored
6. Then create the missing/additive Graphify outputs in the correct locations.

Required output format:
1. Repo source-root understanding
2. Existing Graphify inventory
3. Usable for ContextBridge
4. Missing for ContextBridge
5. Additive files/folders to create
6. Risks or assumptions
7. Final implementation summary

Important constraints:
- Do not delete existing Graphify data.
- Do not replace existing files unless explicitly required.
- Prefer additive normalization.
- If existing Graphify data is structurally different but still useful, adapt around it instead of forcing a rewrite.
- If ownership graphs are missing, create them.
- If business packs are missing source-files.txt or raw/graph.json, add them.
- When creating or completing source-files.txt, include ALL layers of the stack:
    controller, service, entity/model, DTO, UI page/component.
  Service files are the most commonly omitted and the most important for CB routing.
  A pack missing its service file means CB cannot route to the root cause.
- Also include runtime and bridge ownership when relevant to the flow:
    job/scheduler, resolver, policy, middleware, filter, bridge/handoff, router/action-panel,
    print/export controller/API, and downstream posting/runtime consumer files.
  A pack that contains only setup/config files but not the runtime consumer path is incomplete
  for ContextBridge debugging.
- If cross-file dependency edges are weak, add enrichment files instead of editing original graph outputs first.
- If the right files are present but the pack still favors symptom files over owner files,
  add an additive retrieval-hints behavior file that explicitly maps:
  - real owner files
  - runtime owner files
  - bridge/handoff files
  - real-world business/symptom query terms
- Ensure the primary owner files named in `behavior/retrieval-hints.md` stay aligned
  with `raw/source-files.txt`. Do not describe one owner path in retrieval hints while
  `source-files.txt` is centered on a different owner path.
- Always add retrieval-hints for shell/view/runtime owner patterns when those
  files exist but are not likely to win from raw source-files alone:
  - admin configuration / audit / permissions pages for operational modules
  - dashboard shell/layout/global-search entry
  - legacy shell / overlay / connection-state UI
  - drawer/runtime/payment-side POS UI
  - workspace context/layout/view owners
  - startup/bootstrap/app-entry/runtime provider owners
  - reports/analytics/audit pages
  - receipt-template designer/list/editor pages
  - organization/store/terminal settings pages
  - investigation file preview/highlight/delete APIs
  - service catalog / pricing / billable charge APIs
- If a pack represents admin operations or runtime configuration, the retrieval
  hints must name the admin API/page/permission owners for that pack. Do not let
  retrieval hints drift toward sibling runtime/workflow owners that are not the
  pack's real primary path.
- Also include symptom wording that may omit the top-level module name entirely,
  such as `event images` + `linked order details`, `treatment master` + `staff availability`,
  or `report guard` + `exception filter`.

Normalization decision rule:
- If the miss is "file absent from pack", fix `raw/source-files.txt`.
- If the miss is "owner present but not discoverable from real bug wording", fix behavior docs with additive retrieval hints.
- If the miss is "cross-module path missing explicit runtime edge", add enrichment rather than replacing original graph outputs.

Success criteria:
- ContextBridge should be able to discover the normalized Graphify structure.
- Central packs should support business/workflow understanding.
- Folder-level ownership graphs should support owner-file and dependency tracing.
- Existing Graphify investment should be preserved as much as possible.
