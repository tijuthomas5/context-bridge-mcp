Role: Graphify Bootstrap Architect for ContextBridge

Objective:
This repository has no Graphify data yet. Inspect the codebase and create the initial Graphify structure required for ContextBridge to work effectively.

Context:
There is currently no usable Graphify output in this repository. Your job is to design and generate the first-pass Graphify structure so ContextBridge can support:
- business/workflow discovery
- owner-file discovery
- dependency tracing
- API/service/controller/UI mapping

Important rule:
Do not assume folder names such as `main_ui`, `main_service`, `src`, `client`, `server`, `frontend`, or `backend`.

You must first inspect the repo and infer:
- frontend/UI source roots
- backend/service/API source roots
- shared/core/common roots
- module/vertical roots
- important domain areas
- likely integration/cross-module areas

If these boundaries are unclear after inspection, stop and ask the user for:
- frontend root(s)
- backend root(s)
- shared/core root(s)
- preferred Graphify root location

What must be created:
1. A central Graphify root such as:
   graphify-out/

2. Business/flow packs under module folders such as:
   graphify-out/<module>/<pack>/behavior/
   graphify-out/<module>/<pack>/raw/source-files.txt
   graphify-out/<module>/<pack>/raw/graph.json

3. Folder-level ownership graphs near source areas where relevant:
   <source-area>/graphify-out/graph.json

CRITICAL — single source of truth per file:
A file that belongs to both a business/flow pack (item 2) and a folder-level ownership
graph (item 3) will end up with a symbol-level node list in more than one graph.json.
Derive that file's node list ONCE, then copy the same node/edge entries identically into
every graph.json that references this file. Do not independently re-derive or re-walk the
same file's symbols separately for each location -- doing so lets the copies drift apart
over time (one copy captures a declaration the other misses, with no way to tell). If you
update or add a node for a file in one graph.json, apply the identical update to every
other graph.json that also references that file, in the same pass.

What ContextBridge needs from Graphify:
- business packs for workflow understanding
- source-files.txt for exact file routing
- raw/graph.json for structural graph retrieval
- ownership graphs for controller/service/entity/API/UI tracing
- optional enrichment files for explicit dependency chains when needed

CRITICAL — source-files.txt completeness rule:
Every source-files.txt must include ALL layers of the stack for that pack:
  - controller file(s)
  - service file(s)  ← most commonly omitted, most important for CB routing
  - entity/model file(s)
  - DTO file(s)
  - UI page/component file(s)
Do NOT list only entities and DTOs. A pack missing its service file means CB cannot
route queries to the root cause — it will find symptom files instead.

CRITICAL — runtime and bridge completeness rule:
If the workflow has runtime execution or cross-module handoff, source-files.txt must
also include the real runtime/bridge owner files where relevant:
  - job / scheduler files
  - resolver / policy / middleware / filter files
  - bridge / handoff files between modules
  - workspace/router/action-panel entry files
  - print/export controller or API files
  - downstream posting / allocation / ledger / billing owners
Do NOT stop at setup/configuration files when the real bug path runs through runtime consumers.

CRITICAL — cross-layer UI-to-backend edge rule:
Listing a UI API-call file and its backend controller/route file in the same
source-files.txt is not enough. When building graph.json for a pack that spans both
layers, explicitly create an edge between the UI call site and the backend controller
method it invokes, matched by route path + HTTP verb (not by naming similarity alone).
A pack where the frontend and backend files are merely co-listed but have no connecting
edge in graph.json is not fully wired -- ContextBridge cannot traverse from the UI file
to its backend owner without that edge. If the match is ambiguous in V1, it is
acceptable to skip the edge, but note it explicitly in the pack's output rather than
silently omitting it.

Wrapper vs. page component: if the file that actually makes the route+verb call is an
API-wrapper/client file (not the page/component itself), first check whether an import
edge already connects the page component to that wrapper. If it does, the route+verb
edge only needs to land on the wrapper -- the component is already transitively
reachable through the existing import edge, so do not duplicate the route+verb edge onto
the component. If no import edge connects the component to the wrapper, derive that
import edge as well, so the component-to-backend chain is actually complete and not just
assumed.

Your tasks:
1. Inspect the repository structure.
2. Identify source roots and major modules.
3. Identify the most important functional/business flows.
4. Propose the initial Graphify pack structure.
5. Propose the folder-level ownership graph locations.
6. Create the first-pass Graphify outputs in the correct structure.
7. Keep the first pass practical and high ROI. Do not try to graph every corner of the repo at once.

First-pass prioritization rules:
- Start with the modules and flows most likely to be queried by AI.
- Prefer high-value flows such as:
  - core CRUD/business workflows
  - approval flows
  - billing/payment/order flows
  - notification/integration flows
  - dashboard/search/report flows
  - UI -> API -> controller -> service -> entity chains
  - runtime config consumer chains
  - cross-module handoff chains
  - workspace shell -> action panel -> workflow entry chains
- Add folder-level ownership graphs for the main source areas first.
- Do not attempt deep AST-level graphing for every file in V1.
- Whatever symbol-level nodes you DO include for a file in graph.json, do not capture
  them inconsistently -- do not record some of a file's top-level functions/const-
  functions/arrow-functions/classes as nodes while silently skipping others of the same
  shape. It is fine to keep V1 shallow (fewer files, fewer symbols), but within any file
  you do choose to graph at the symbol level, be complete for that file rather than
  partially and silently complete. A dedicated later pass (prompt 1.4 in this same
  folder) exists specifically to close symbol-level gaps in files that turn out to need
  deeper coverage -- V1 does not need to solve that, it only needs to avoid creating
  silent, undetectable partial coverage.

For every first-pass pack, also create a small behavior note that helps retrieval
from real developer wording. Prefer an additive file such as:
  behavior/retrieval-hints.md

That file should name:
- the real owner files
- the runtime/bridge files
- the business/symptom terms developers actually use in bug reports
- the common false neighbors that should not dominate the pack
- the same primary owner path that is represented in `raw/source-files.txt` for that pack
- For shell/view/runtime owner packs, retrieval-hints are not optional. Create
  them in the first pass for:
  - admin configuration / audit / permissions pages for operational modules
  - dashboard shell/layout/global-search entry
  - legacy shell / overlay / connection-state UI
  - drawer/runtime/payment-side POS UI
  - workspace context/layout/view owners
  - startup/bootstrap/app-entry/runtime provider owners
  - reports/analytics/audit pages
  - receipt-template designer/list/editor pages
  - organization/store/terminal settings pages
  - investigation file preview/highlight/delete APIs
  - service catalog / pricing / billable charge APIs
- For admin-operation or runtime-configuration packs, make the retrieval hints
  name the admin API/page/permission owners explicitly. Do not point those hints
  at sibling runtime/workflow files unless those files are also primary owners in
  `raw/source-files.txt`.
- Include real bug wording even when the module name is omitted, for example:
  `event images` + `linked order details`, `treatment master` + `staff availability`,
  `report guard` + `exception filter`.
- Cross-pack backend ownership check: when a pack's workflow ends in a state change owned
  by a shared/central service that actually belongs under a different pack (e.g. a charge,
  pricing, or ledger-posting service that naturally sits in a "catalog"/"pricing"/"billing"
  pack but is genuinely invoked by this pack's own workflow), do not treat the workflow
  pack as self-contained. Include that shared owner file in this pack's own
  `raw/source-files.txt` too -- a workflow's true decisive owner can legitimately belong to
  more than one pack's file list from the start, not just get added later as a fix.
  CRITICAL -- do not stop at the `raw/source-files.txt` line. Run this shared owner file
  through the same real AST/graph extraction as every other file in this pack, so it gets
  real class/method nodes in this pack's `raw/graph.json` and a copy in `raw/corpus/`.
  ContextBridge's ranking gives real scoring weight to actual graph presence, not to a
  plain text mention -- a file only listed in `source-files.txt` stays effectively
  invisible to ranking even though it is technically "in" the pack. Build this in from
  the first pass, the same way every other pack file gets real extraction.
- Explicit-decisive-owner check: when a pack's `behavior/retrieval-hints.md` will list
  several plausible owner files that could all reasonably answer a similar-sounding query
  (e.g. several UI modal files, or a generic service next to the one file that actually
  performs the specific action), do not just list them side by side. Name which one is THE
  decisive owner for the pack's core symptom, and say briefly why (e.g. "X posts the actual
  charge; Y and Z only read/display it"). Build this distinction in from the first pass
  rather than leaving all candidates undifferentiated.

Required output format:
1. Repo source-root understanding
2. Proposed Graphify structure
3. Central pack plan
4. Folder-level ownership graph plan
5. Files/folders to create
6. Risks or assumptions
7. Final implementation summary

Important constraints:

- Optimize for ContextBridge usefulness, not theoretical completeness.

Operational safety rules:

  - test files and test folders such as `__tests__`, `*.test.*`, `*.spec.*`
  - vendor/build outputs such as `node_modules`, `dist`, `build`, `bin`, `obj`
  - nested Graphify artifacts, backup Graphify folders, and temporary Graphify build folders if they already exist in parts of the repo
- Build new Graphify outputs in isolated temporary locations first when practical, then move the finished `graphify-out` into place only after validating that required outputs exist such as `graph.json`, `GRAPH_REPORT.md`, `graph.html`, and `manifest.json` when applicable.
- If the chosen scope includes docs, images, or papers and Graphify blocks on missing semantic-extraction API keys, do not leave the repo half-updated. Either narrow the scope to code-only for the first pass or stop and report the blocker clearly.
- After creating initial Graphify outputs, explicitly remind the user that ContextBridge must be reindexed or synced before CB can use the new data.

Success criteria:
- The repo ends up with a usable Graphify structure for ContextBridge.
- ContextBridge can route by business pack and by ownership graph.
- The first-pass Graphify layout is scalable and can be enriched later.
- The output is practical for a real developer workflow, not just documentation-heavy.
