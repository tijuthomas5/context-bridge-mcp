Role: Graphify Bootstrap Architect for ContextBridge

Objective:
This repository has no Graphify data yet. Inspect the codebase and create the initial Graphify structure required for ContextBridge to work effectively.

Context:
There is currently no usable Graphify output in this repository. Your job is to design and generate the first-pass Graphify structure so ContextBridge can support:
- business/workflow discovery
- owner-file discovery
- dependency tracing
- API/service/controller/UI mapping

Important rule:
Do not assume folder names such as `main_ui`, `main_service`, `src`, `client`, `server`, `frontend`, or `backend`.

You must first inspect the repo and infer:
- frontend/UI source roots
- backend/service/API source roots
- shared/core/common roots
- module/vertical roots
- important domain areas
- likely integration/cross-module areas

If these boundaries are unclear after inspection, stop and ask the user for:
- frontend root(s)
- backend root(s)
- shared/core root(s)
- preferred Graphify root location

What must be created:
1. A central Graphify root such as:
   graphify-out/

2. Business/flow packs under module folders such as:
   graphify-out/<module>/<pack>/behavior/
   graphify-out/<module>/<pack>/raw/source-files.txt
   graphify-out/<module>/<pack>/raw/graph.json

3. Folder-level ownership graphs near source areas where relevant:
   <source-area>/graphify-out/graph.json

What ContextBridge needs from Graphify:
- business packs for workflow understanding
- source-files.txt for exact file routing
- raw/graph.json for structural graph retrieval
- ownership graphs for controller/service/entity/API/UI tracing
- optional enrichment files for explicit dependency chains when needed

CRITICAL — source-files.txt completeness rule:
Every source-files.txt must include ALL layers of the stack for that pack:
  - controller file(s)
  - service file(s)  ← most commonly omitted, most important for CB routing
  - entity/model file(s)
  - DTO file(s)
  - UI page/component file(s)
Do NOT list only entities and DTOs. A pack missing its service file means CB cannot
route queries to the root cause — it will find symptom files instead.

CRITICAL — runtime and bridge completeness rule:
If the workflow has runtime execution or cross-module handoff, source-files.txt must
also include the real runtime/bridge owner files where relevant:
  - job / scheduler files
  - resolver / policy / middleware / filter files
  - bridge / handoff files between modules
  - workspace/router/action-panel entry files
  - print/export controller or API files
  - downstream posting / allocation / ledger / billing owners
Do NOT stop at setup/configuration files when the real bug path runs through runtime consumers.

Your tasks:
1. Inspect the repository structure.
2. Identify source roots and major modules.
3. Identify the most important functional/business flows.
4. Propose the initial Graphify pack structure.
5. Propose the folder-level ownership graph locations.
6. Create the first-pass Graphify outputs in the correct structure.
7. Keep the first pass practical and high ROI. Do not try to graph every corner of the repo at once.

First-pass prioritization rules:
- Start with the modules and flows most likely to be queried by AI.
- Prefer high-value flows such as:
  - core CRUD/business workflows
  - approval flows
  - billing/payment/order flows
  - notification/integration flows
  - dashboard/search/report flows
  - UI -> API -> controller -> service -> entity chains
  - runtime config consumer chains
  - cross-module handoff chains
  - workspace shell -> action panel -> workflow entry chains
- Add folder-level ownership graphs for the main source areas first.
- Do not attempt deep AST-level graphing for every file in V1.

For every first-pass pack, also create a small behavior note that helps retrieval
from real developer wording. Prefer an additive file such as:
  behavior/retrieval-hints.md

That file should name:
- the real owner files
- the runtime/bridge files
- the business/symptom terms developers actually use in bug reports
- the common false neighbors that should not dominate the pack
- the same primary owner path that is represented in `raw/source-files.txt` for that pack
- For shell/view/runtime owner packs, retrieval-hints are not optional. Create
  them in the first pass for:
  - admin configuration / audit / permissions pages for operational modules
  - dashboard shell/layout/global-search entry
  - legacy shell / overlay / connection-state UI
  - drawer/runtime/payment-side POS UI
  - workspace context/layout/view owners
  - startup/bootstrap/app-entry/runtime provider owners
  - reports/analytics/audit pages
  - receipt-template designer/list/editor pages
  - organization/store/terminal settings pages
  - investigation file preview/highlight/delete APIs
  - service catalog / pricing / billable charge APIs
- For admin-operation or runtime-configuration packs, make the retrieval hints
  name the admin API/page/permission owners explicitly. Do not point those hints
  at sibling runtime/workflow files unless those files are also primary owners in
  `raw/source-files.txt`.
- Include real bug wording even when the module name is omitted, for example:
  `event images` + `linked order details`, `treatment master` + `staff availability`,
  `report guard` + `exception filter`.

Required output format:
1. Repo source-root understanding
2. Proposed Graphify structure
3. Central pack plan
4. Folder-level ownership graph plan
5. Files/folders to create
6. Risks or assumptions
7. Final implementation summary

Important constraints:
- Do not invent fake architecture.
- Read the repo first.
- Do not assume module boundaries without evidence.
- Keep Graphify outputs additive and organized.
- Optimize for ContextBridge usefulness, not theoretical completeness.
- If a flow is too broad, split it into smaller packs.
- If exact dependencies are not obvious, still create the first-pass ownership graphs and source file maps.

Success criteria:
- The repo ends up with a usable Graphify structure for ContextBridge.
- ContextBridge can route by business pack and by ownership graph.
- The first-pass Graphify layout is scalable and can be enriched later.
- The output is practical for a real developer workflow, not just documentation-heavy.
