Role: Graphify Validation Architect for ContextBridge

Objective:
This repository already has Graphify data and it is expected to already follow the ContextBridge-friendly structure. Your job is to validate that assumption, identify any remaining gaps, and improve only what is necessary.

Context:
Graphify is already present in this repository. It may already be correctly structured for ContextBridge, but that must be verified from the actual files and folders, not assumed.

Important rule:
Do not assume folder names such as `main_ui`, `main_service`, `src`, `client`, `server`, `frontend`, or `backend`.

You must first inspect the repo and infer:
- frontend/UI source roots
- backend/service/API source roots
- shared/core/common roots
- module/vertical roots
- Graphify roots

If these boundaries are unclear after inspection, stop and ask the user for:
- frontend root(s)
- backend root(s)
- shared/core root(s)
- Graphify root(s)

What ContextBridge expects:
1. A central Graphify root such as:
   graphify-out/

2. Business/flow packs under module folders such as:
   graphify-out/<module>/<pack>/behavior/
   graphify-out/<module>/<pack>/raw/source-files.txt
   graphify-out/<module>/<pack>/raw/graph.json

3. Folder-level ownership graphs near source areas where relevant:
   <source-area>/graphify-out/graph.json

4. Optional additive enrichments where needed:
   enrichment/dependency-edges.enrichment.json

Your tasks:
1. Inspect the repository structure.
2. Inspect all existing Graphify folders and files.
3. Validate whether the Graphify structure already matches ContextBridge needs.
4. Classify the result into:
   - fully usable
   - usable with gaps
   - structurally incomplete
5. Report:
   - what is already correct
   - what is missing
   - what is weak
   - what should be improved
   - what should be ignored
6. Only create or update additive outputs if needed.
7. Do not rewrite or replace existing Graphify data unless explicitly required.

Validation rules:
- Do not assume that Graphify existing means Graphify is sufficient.
- Check whether packs actually contain:
  - behavior docs
  - raw/source-files.txt
  - raw/graph.json
- For each source-files.txt that exists, verify it includes ALL layers of the stack:
    controller, service, entity/model, DTO, UI page/component.
  Service files are the most commonly omitted. A pack missing its service file means
  CB cannot route queries to the root cause — it will find symptom files instead.
  If service files are missing from source-files.txt, add them as an additive fix.
- Do not stop at controller/service/entity/DTO/UI only. If the workflow depends on
  runtime execution or cross-module handoff, also verify the pack includes the
  true runtime owner files such as:
  - job / scheduler files
  - resolver / policy / middleware / filter files
  - bridge / handoff files between modules
  - print/export controller or API files
  - workspace/action-panel/router entry files when the workflow is shell-driven
  If these files are part of the real debugging path but missing from the pack,
  add them as additive fixes.
- If the pack already contains the right files but weak queries still drift to
  adjacent files, verify whether the behavior layer contains explicit retrieval
  wording for the workflow. If not, add an additive behavior note such as
  `behavior/retrieval-hints.md` that names:
  - the real owner files
  - the runtime/bridge files
  - the business/symptom terms developers actually use in bug reports
- Ensure the primary owner files named in `behavior/retrieval-hints.md` stay aligned
  with `raw/source-files.txt`. Do not name unrelated owners in retrieval hints if the
  pack's actual source-files already point to a different owner path.
- Treat the following as mandatory retrieval-hints cases when the files already
  exist but retrieval is weak:
  - admin configuration / audit / permissions pages for operational modules
  - dashboard shell/layout/global-search entry
  - legacy shell / overlay / connection-state UI
  - drawer/runtime/payment-side POS UI
  - workspace context/layout/view owners
  - startup/bootstrap/app-entry/runtime provider owners
  - reports/analytics/audit pages
  - receipt-template designer/list/editor pages
  - organization/store/terminal settings pages
  - investigation file preview/highlight/delete APIs
  - service catalog / pricing / billable charge APIs
- If a pack is about admin operations or runtime configuration, make sure
  `behavior/retrieval-hints.md` names the admin API/page/permission owners for that
  pack, not only nearby runtime or workflow files from sibling modules.
- In retrieval hints, include real bug wording even when the top-level module word is
  omitted. Example patterns: `event images` + `linked order details`, `treatment master`
  + `staff availability`, `report guard` + `exception filter`.
- Cross-pack backend ownership check: when a pack's workflow ends in a state change owned
  by a shared/central service that actually lives in a different pack (e.g. a charge,
  pricing, or ledger-posting service defined under a "catalog"/"pricing"/"billing" pack
  but genuinely invoked by this pack's own workflow), do not assume the workflow pack is
  self-contained. Confirm whether that shared owner file is present in this pack's own
  `raw/source-files.txt`. If it is missing, add it here as an additive fix even though its
  primary home is a different pack -- a workflow's true decisive owner can legitimately
  belong to more than one pack's file list. Flag this explicitly as a missing-file gap,
  not a wording gap.
  CRITICAL -- a `raw/source-files.txt` line alone is NOT a sufficient fix for this case.
  ContextBridge's ranking gives real scoring weight to a file's actual graph presence
  (`raw/corpus/` + nodes/edges in `raw/graph.json`), not to a plain text mention -- a
  file listed only in `source-files.txt` stays effectively invisible to ranking even
  though it is technically "in" the pack. When this check finds a missing shared owner,
  the fix must run that file through the same real AST/graph extraction this pack's other
  files went through (so it gets its own class/method nodes in `raw/graph.json` and a
  copy in `raw/corpus/`), not just an added line in `source-files.txt`. Only fall back to
  a text-only listing if real extraction genuinely cannot be run, and say so explicitly
  when reporting the fix.
- Explicit-decisive-owner check: when a pack's `behavior/retrieval-hints.md` lists several
  plausible owner files that could all reasonably answer a similar-sounding query (e.g.
  several UI modal files, or a generic ledger/forms service next to the one file that
  actually performs the specific action), do not just list them side by side. Name which
  one is THE decisive owner for the pack's core symptom, and say briefly why (e.g. "X
  posts the actual charge; Y and Z only read/display it"). A hint file that lists five
  plausible files with no distinction between them is a weaker fix than one that lists the
  same five files but marks the real decisive owner.
- Check whether ownership graphs exist near source areas.
- Symbol-level completeness check: for a small sample of files already covered by a
  graph.json (not the whole repo), count the top-level functions/const-functions/
  arrow-functions/classes actually declared in the real source file and compare that
  against the nodes graph.json has for that same file. Do this for at least one file
  per pack you validate. A file can be inconsistent even when the pack/ownership
  structure is otherwise correct -- some declarations get silently skipped while
  others of the same shape are captured, with no error or warning. If a sampled file
  shows a mismatch, flag the pack as "usable with gaps" rather than "fully usable,"
  and note it as a candidate for a dedicated symbol-completeness pass (see prompt 1.4
  in this same folder) rather than trying to fix every symbol here.
- Also check whether the same file is referenced from more than one graph.json (a
  pack-level graph and a module/ownership-level graph both covering it). If so, note
  whether their node lists actually agree -- one copy having a symbol the other lacks
  is the same kind of gap and should be flagged the same way.
- CRITICAL -- single source of truth per file: this kind of copy-vs-copy disagreement
  happens because each graph.json independently re-derives the same file's symbols
  instead of sharing one derivation. When you additively fix a mismatch you find here,
  apply the identical fix to every graph.json copy of that file in the same pass --
  never fix just the one copy you happened to check first.
- Cross-layer edge check: if a pack contains both frontend API-call files (e.g. UI
  files that call an endpoint) and the matching backend controller/route files, do not
  treat their co-presence in `raw/source-files.txt` as sufficient. Check whether
  `graph.json` actually contains an edge between the UI call site and the backend
  controller method (matched by route path + HTTP verb, not just naming similarity).
  If both files exist in the pack but no such edge exists in the graph, flag it as a
  missing cross-layer edge -- this is the same class of gap as a missing owner file,
  just one layer higher (file present, connection absent). Do not fix it inline here;
  note it as a candidate for an additive `enrichment/dependency-edges.enrichment.json`
  fix, applied the same way normalization prompt 1.2 handles other enrichment gaps.
  Walk the full chain, not just the nearest file: when the route+verb call is actually
  made from an API-wrapper/client file rather than the page/component itself, do not
  mark the pack "connected" just because the wrapper has a backend edge. Also confirm
  an import edge connects the page component to that wrapper. Only if both links exist
  (component -> wrapper -> backend) is the pack actually wired end to end. A wrapper
  that is linked while its calling component has no import edge to it means the
  component is still effectively isolated for primary-owner routing purposes, even
  though the wrapper "passes" the check -- flag that case separately from a fully
  missing edge, since the fix differs (add the component->wrapper import edge, not
  another route+verb edge).
- Check whether the data is useful for:
  - business/workflow routing
  - owner-file routing
  - dependency tracing
  - UI/API/controller/service/entity mapping
- Validate owner completeness for each important pack:
  - root mutation/state owner
  - runtime consumer
  - bridge/handoff owner
  - UI/API entry owner
  If only symptom/shell/history files are present, mark the pack as weak even if
  Graphify files technically exist.

If Graphify is already good enough:
- do not make unnecessary changes
- only report readiness and minor recommendations

If Graphify has small gaps:
- fill only those gaps
- prefer additive files
- prefer enrichment over replacement

Required output format:
1. Repo source-root understanding
2. Existing Graphify inventory
3. Validation result
4. Already usable for ContextBridge
5. Missing or weak areas
6. Additive improvements to make
7. Risks or assumptions
8. Final implementation summary

Important constraints:
- Do not delete existing Graphify data.
- Do not replace existing files unless explicitly required.
- Prefer additive improvements.
- Preserve existing Graphify investment.
- Optimize for ContextBridge usefulness, not theoretical completeness.

Operational safety rules:
- Exclude non-source and polluting inputs from Graphify work unless the user explicitly asks for them:
  - test files and test folders such as `__tests__`, `*.test.*`, `*.spec.*`
  - vendor/build outputs such as `node_modules`, `dist`, `build`, `bin`, `obj`
  - nested Graphify artifacts such as `graphify-out`, backup Graphify folders, and temporary Graphify build folders
- Treat backup or temporary Graphify folders such as `graphify-out_pre_*`, `graphify-out_live_before_*`, and `_graphify_*` as non-indexable noise for ContextBridge purposes. Do not graph them and do not leave them inside the effective scan set for a rebuild.
- When an existing Graphify folder must be improved, prefer a safe rebuild pattern:
  1. create a backup of the current live Graphify folder
  2. build the new Graphify output in an isolated temporary folder OUTSIDE the pack's own
     `raw/` directory entirely (e.g. a sibling temp path, not a subfolder nested inside
     `raw/`) -- a temp build folder created INSIDE `raw/` is still inside ContextBridge's
     scan path even though it feels "isolated," and a folder literally named `graphify-out`
     nested inside `raw/` will get indexed as if it were a second, separate pack. Real
     diagnosed case: a rebuild wrote its output to `raw/graphify-out/` instead of a true
     external temp location, and that leftover nested folder silently double-counted every
     file it covered in real search results because nothing in step 4 ever removed it.
  3. validate that required outputs exist such as `graph.json`, `GRAPH_REPORT.md`, `graph.html`, and `manifest.json` when applicable
  4. replace the live Graphify folder only after the isolated build succeeds, THEN delete
     the temporary build folder from step 2 -- do not leave it behind "just in case." A
     leftover temp build folder with its own `graph.json`/`manifest.json` inside the scan
     path is functionally identical to a duplicate pack and will get indexed.
- If Graphify fails because docs, images, or papers require semantic extraction and no API key is available, do not damage the current output. Fall back to a code-only rebuild when that still satisfies the user goal, and report clearly that the rebuild excluded semantic doc/image extraction.
- After any Graphify create/update/normalization work, explicitly remind the user that ContextBridge must be reindexed or synced before CB can use the new Graphify data.

Success criteria:
- The repository’s Graphify structure is validated against ContextBridge needs.
- Any missing but high-value gaps are identified clearly.
- If needed, only minimal additive improvements are made.
- Existing correct Graphify data remains intact.
