# ContextBridge — Generate Ranking Rules JSON
### Run this AFTER generating the project profile (2.GENERATE_PROJECT_PROFILE.md)

---

## WHAT IS THE RULES FILE AND WHY DO YOU NEED IT

`rules/projects/<projectname>_rules.json` is a simple JSON boost/penalty file that works alongside the profile.

The profile (`_profile.py`) handles complex domain logic — intent detection, scoring formulas, token expansion.
The rules file handles **simple one-off corrections** — "always boost file X and penalize file Y when query contains Z."

Running this prompt before first use gives you better retrieval quality from day one instead of waiting for misses to appear.

---

## STEP 1 — GIVE THE AI THIS PROMPT

Give the AI access to your codebase and your generated `_profile.py`, then send the prompt below.

---

## THE PROMPT (copy from here)

```
You are generating a ContextBridge ranking rules file for this codebase.

This file fine-tunes file ranking for specific queries using simple boost and penalty rules.
It works alongside the existing _profile.py — do not duplicate what the profile already handles.

Output file: rules/projects/<projectname>_rules.json

---

STEP 1 — READ THE EXISTING PROFILE

Read rules/projects/<projectname>_profile.py and identify:
- Which modules and intent detectors already exist
- Which files are already pinned or boosted in adjust_owner_score
- Which files are already suppressed

Do NOT add rules.json entries for files the profile already handles well.

---

STEP 2 — IDENTIFY RANKING CONFLICT PAIRS

For each module, identify pairs where:
- File A is the true root-cause owner
- File B is a generic/shared file that often ranks above A incorrectly

Examples of common conflicts:
- A broad controller ranks above a specific service
- A UI overview page ranks above the specific feature page
- A shared API client ranks above the feature-specific controller
- A registration/list page ranks above the detail/edit page
- A workspace root ranks above a sub-feature page

List at least 3-5 conflict pairs per major module.

Also identify **global noise files** — files that rank high across ALL modules due to
generic keywords (e.g. auth/permission/user/role files, base controllers, shared utilities).
These should be penalized in rules where they have no ownership stake.

Also identify **narrative distractor terms** — words common in incident reports or tickets
(e.g. "legal", "supervisor", "family", "team", "critical", "blocked") that keyword-match
generic files instead of the true domain owners. Rules targeting incident-style queries
should use specific domain terms (method names, state names, entity names), not story words.

Also identify **intra-module intent conflicts** — cases where two intents within the same
module can fire simultaneously on the same query (e.g. a fulfillment bug report that mentions
"checkout is blocked" fires both a checkout intent and a fulfillment intent). rules.json
path penalties alone CANNOT fix intra-module intent conflicts — the wrong files arrive
through pins and score multipliers that rules.json cannot override. If you identify this
pattern, flag it as requiring a competing-signal guard in the profile (_profile.py), not
a rules.json entry.

Also identify **runtime-owner gaps that rules.json must NOT paper over**:
- missing drawer/session/policy/runtime resolver owners
- billing/ledger/account-state queries routing to generic billing files
- package consumption/runtime queries routing to package setup files
- notification/follow-up runtime queries routing to admin notification shell files
- dashboard shell/layout/global-search queries routing only to backend dashboard providers
- legacy POS shell / sync / overlay queries routing only to POS services
- POS drawer/runtime UI queries routing only to cashiering admin tabs or backend movement files
- customer workspace view/layout queries routing only to billing/history pages
- bootstrap/startup/runtime queries routing only to downstream cart/sales services
- receipt template editor/designer queries routing only to print-profile backend files
- settings/runtime admin queries routing only to unrelated cashiering/runtime files
These are profile problems first. Only add rules.json boosts after the profile already
pins and boosts the correct runtime owners.
If the conflict is caused by ambiguous words like `terminal` or `runtime` appearing in both
an admin-settings query and a cashiering/POS-runtime query, treat it as a profile competing-intent
problem first. Do not patch that drift with rules.json.
If the miss is caused by a broad intake/booking workflow outranking a narrower resource-space /
reservation / occupancy workflow, treat that as a profile competing-intent problem first.
Do not patch it with rules.json.
If `runtime` or `configuration` alone is causing POS/bootstrap shell files to appear for an
unrelated workflow, treat that as a profile intent-definition problem first.

Before creating any rule, classify the miss as one of:
- simple ranking conflict
- project profile issue
- graph/index coverage issue

If the benchmark expects exact UI shell/view/runtime owners and retrieval is returning only nearby backend files,
classify that as a profile issue first unless the profile already pins and boosts those exact UI owners.

If the query uses explicit ownership wording such as `API`, `backend components`, `controller`, or `service`,
and the profile does not route/boost the backend owners for that wording, classify that as a profile issue first.

If Graphify has a dedicated pack for the subdomain but `_profile.py` does not wire that subdomain into
`pack_files_for_intents`, classify that as a profile issue first, not a rules.json gap.

Only simple ranking conflicts belong in rules.json.
If the miss is caused by missing subdomain routing, missing runtime-consumer owners, missing bridge/handoff owners,
or weak owner boosting, fix `_profile.py` instead of adding a rules.json band-aid.

---

STEP 3 — IDENTIFY QUERY-SPECIFIC BOOSTS

For each conflict pair from Step 2, identify:
- What words would a developer use in a query that should trigger this rule?
- Which file should rank higher (path_boosts)?
- Which file should rank lower (path_penalties)?

Use specific query terms, not generic ones. A rule that fires on "order" alone is too broad.
A rule that fires on ["order", "cancel", "status"] is precise.

If the query depends on:
- runtime consumer vs config editor
- bridge / handoff between modules
- same-module subdomain routing
- owner completeness across backend + runtime + bridge
do NOT solve it with a rules.json entry unless the profile already returns the correct owners and only the final ordering is slightly off.

---

STEP 4 — GENERATE THE RULES FILE

Write the complete rules JSON file with this structure:

{
  "profile": "<projectname>",
  "semantic_rules": [
    {
      "name": "<descriptive_rule_name>",
      "query_profiles": ["code_debug"],
      "terms_any": ["word1", "word2", "word3"],
      "path_boosts": [
        { "suffix": "<filename_lowercase.ext>", "score": 8.0 }
      ],
      "path_penalties": [
        { "suffix": "<filename_lowercase.ext>", "score": 5.0 }
      ]
    }
  ]
}

Rules:
- "name": short descriptive slug (e.g. "order_cancel_service_boost")
- "query_profiles": default to ["code_debug", "cross_module_bridge"] for all new rules. Add "infra_runtime" for infrastructure queries. NEVER use only ["code_debug"] — any query that spans two modules (e.g. "service session blocks checkout ledger") is classified as cross_module_bridge and rules with only code_debug are silently skipped for those queries.
- "terms_any": 2-4 specific words — rule fires if ANY of these appear in the query. Use domain terms (entity names, state names, method names), NOT narrative words (blocked, critical, team, supervisor). IMPORTANT: each entry must be a single word, never a multi-word phrase — the engine tokenizes queries word-by-word, so "direct sign" never matches; use "signed" instead.
- "terms_min_any": optional integer — minimum number of terms_any that must match before rule fires. Use 2 when terms_any contains short/common words to avoid false fires. Omit when terms are already specific enough.
- "path_boosts": suffix must match the end of the normalized (lowercase) file path
- "path_penalties": same suffix format. Use penalties aggressively on global noise files (auth, permission, role, user controllers) when those files have no ownership stake in the query domain.
- score range: 3.0 (mild) to 12.0 (strong) — use higher scores for clearly unique owner files
- Do NOT add a rule if the profile already handles it

---

STEP 5 — VERIFY YOUR OUTPUT

Before finishing, check:
[ ] Every major module has at least 2-3 rules
[ ] No rule duplicates what _profile.py already does
[ ] Every rule has "cross_module_bridge" in query_profiles (not just "code_debug")
[ ] Every rule has at least 2 terms_any entries (not just 1 generic word)
[ ] terms_any uses domain/technical terms, not narrative/incident words
[ ] Global noise files (auth, permission, role, user) are penalized in rules where they don't belong
[ ] terms_min_any is set to 2 on any rule with short or common terms_any words
[ ] path_boosts and path_penalties use lowercase file name suffixes
[ ] Scores are proportional — primary owner file gets highest score
[ ] Rule names are descriptive and unique
[ ] At least 10 rules total covering the whole application
[ ] Every rule was checked to confirm it is a simple ranking conflict, not a profile-routing or graph/index coverage problem
[ ] No rule is being used to patch a missing runtime-consumer owner, missing bridge/handoff owner, or missing subdomain intent

---

SUCCESS CRITERIA
- True owner files rank in top 1-3 for their module-specific queries
- Generic shared files do not dominate feature-specific queries
- No rule is so broad it fires on unrelated queries

OUTPUT: One complete JSON file ready to save as rules/projects/<projectname>_rules.json
```

---

## AFTER THE AI GENERATES THE RULES FILE

1. Save the file to `context_bridge/rules/projects/<projectname>_rules.json`
2. Restart CB using your OS-specific start script:
   - **Windows:** `context_bridge\setup\windows\1.  start_Context_Bridge.bat`
   - **Mac:** `context_bridge/setup/mac/1. start_Context_Bridge.sh`
   - **Linux:** `context_bridge/setup/linux/1. start_Context_Bridge.sh`
3. Run a few test queries to verify ranking improved

---

## WHEN TO UPDATE THIS FILE

- A specific file keeps ranking wrong after real usage
- A new module is added and the profile alone is not enough
- Add a new rule entry — do not delete existing ones unless they are causing harm
