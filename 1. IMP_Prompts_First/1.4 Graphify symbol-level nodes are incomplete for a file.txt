Role: Graphify Symbol Completeness Architect for ContextBridge

Objective:
This repository already has Graphify data (graph.json) for the file(s) named below, and the
pack/ownership structure is already correct. The problem is narrower: graph.json does not
contain a node for every top-level function, const-declared function, arrow function, class,
or method actually defined in the file. Your job is to re-walk the named file(s) and add the
missing symbol-level nodes and their "contains" edge from the file node, without touching
anything else in the file's existing graph.json entry.

Context:
graph.json node capture for this file was done previously (by a prior AI pass or Graphify
run), but it is inconsistent: some top-level declarations were captured as nodes and others
with the *same* syntactic shape were silently skipped. This is not a structural/pack problem
(see 1.1/1.2/1.3) -- the file is already in the right pack, already in source-files.txt, and
the file-level node already exists. The gap is purely at the symbol/node level inside this
one file's contribution to graph.json.

Example of the failure mode this prompt exists to fix (illustrative, any language/file):
  const parseValue = (input) => { ... }                 -> captured as a node
  const buildFilteredRowSet = async ({                  -> NOT captured (same file, same
    optionA, optionB, optionC                              declaration shape, just a
  }) => { ... }                                            longer/multi-line signature)

This kind of gap is not tied to any specific project, language, or naming convention. It has
been observed with plain function declarations, const-assigned arrow functions, async
functions, and multi-line/destructured parameter lists. Do not assume why a declaration was
skipped (signature length, destructuring, async, position in file, file size). Treat every
top-level declaration as required regardless of its shape or complexity.

What you must do:
1. Open the actual file(s) named above. Read the full file, not a summary or an excerpt.
2. Enumerate EVERY top-level declaration in the file:
   - top-level `function name(...) { }` declarations
   - top-level `const name = (...) => { }` / `const name = async (...) => { }` declarations,
     regardless of how many lines the parameter list spans or how it is destructured
   - top-level `class Name { }` declarations and their methods
   - exported and non-exported declarations alike -- do not skip a function just because it
     is not exported
3. Find EVERY graph.json that has an entry for this file -- not just one. The same source
   file is commonly referenced from more than one graph.json in this project: a pack-level
   graph (e.g. graphify-out/<module>/<pack>/raw/graph.json) and a module/ownership-level
   graph (e.g. <source-area>/graphify-out/graph.json) can each independently hold their own
   node list for the same file. Search the repository for every graph.json whose nodes
   contain this file's path as `source_file`, and treat all of them as in scope. Do not
   assume there is only one.
4. For each declaration found, and separately for EACH graph.json located in step 3, check
   whether a node for it already exists (match by normalized label, case-insensitive,
   ignoring a leading "." and a trailing "()"). Build each lookup from that specific
   graph.json's actual current content, not from memory and not from a different copy.
5. For every declaration missing from a given graph.json, add a node to THAT graph.json,
   following the exact schema already used by the existing nodes in that same file (same
   fields: label, file_type, source_file, source_location, id, community, norm_label). Reuse
   the same id/community conventions already present in that graph.json so new nodes are
   indistinguishable in shape from the ones that already exist there.
6. Add a "contains" edge (same relation/shape as existing file -> symbol edges) from the
   file-level node to each newly added symbol node, at the correct source_location line, in
   that same graph.json.
7. Do this consistently across every graph.json found in step 3 -- a declaration that is
   missing in one copy but present in another still counts as missing in the copy that
   lacks it, and must be added there too. Do not stop after fixing the first copy you find.
8. Do not modify, rename, remove, or renumber any existing node or edge in any graph.json.
   This is strictly additive.
9. Do not touch any other file's entry in any graph.json. Scope is limited to the file(s)
   named above.
10. Write each graph.json back as valid JSON, fully re-serialized -- do not leave any
    leftover bytes, padding, or partial writes past the closing brace. Verify the file you
    wrote is parseable JSON before reporting completion.

Self-check before finishing (required, not optional):
- For EACH graph.json found in step 3, re-count the top-level declarations from step 2
  against the nodes now present (existing + newly added) in that specific graph.json. The
  counts must match in every one of them, not just one.
- If any declaration still has no node in any copy, go back and add it before reporting
  completion. Do not report the file as complete while a known declaration is missing a
  node in even one of the graph.json copies.
- Confirm each graph.json you wrote still parses as valid JSON.
- List the declarations you added, with their line numbers, and which graph.json(s) each
  was added to, so the count can be independently verified against the source file.

Required output format:
1. File(s) processed
2. All graph.json copies found for this file (full paths)
3. Declarations found (full list, with line numbers)
4. Per graph.json: nodes that already existed (matched declarations)
5. Per graph.json: nodes added (new declarations that had no node)
6. Per graph.json: edges added
7. Self-check result per graph.json (counts matched: yes/no, JSON still valid: yes/no)
8. Final implementation summary

Important constraints:
- Do not delete or alter any existing node or edge, in any graph.json copy.
- Do not touch pack structure, source-files.txt, behavior docs, or any other file's graph
  data -- this prompt is scoped to symbol-level completeness for the named file(s) only.
- Do not fix only the first graph.json copy you find and stop -- find and fix every copy.
- Do not summarize the file instead of reading it in full -- partial reads are how symbols
  get missed in the first place.
- If the file is very large, it is acceptable to process it in sections, but every section
  must still be read, and the self-check must still cover the whole file before you report
  completion.
- Never leave a graph.json with trailing bytes, null padding, or a partially-written file
  after your edit. Re-serialize the whole JSON document cleanly and confirm it parses.

Success criteria:
- Every top-level function, const-function, arrow function, and class/method in the named
  file(s) has a corresponding node in EVERY graph.json copy that references that file.
- Every new node has a "contains" edge from the file node, in that same graph.json copy.
- The self-check count of declarations-found vs nodes-present matches exactly, in every
  graph.json copy, not just one.
- Every graph.json touched is still valid, parseable JSON after the edit.
- No existing node, edge, or unrelated file was changed.
