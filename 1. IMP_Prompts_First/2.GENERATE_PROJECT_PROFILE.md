# ContextBridge — Auto-Generate Project Profile
### Use this prompt with any AI (Claude, ChatGPT, Gemini, Copilot, etc.)

---

## ⚠️ READ BEFORE YOU START (2 important notices)

**1. Use the most capable mode of your AI.**
When you generate the Graphify data, this CB profile, and the ranking rules/mappings,
use the HIGHEST / smartest mode of whatever model you have (e.g. a "thinking"/"reasoning"
or "pro"/"opus"-tier mode, not the fast/lite mode). This prompt has many cross-checks and
guard rules — a weaker mode tends to skip them, which produces a profile that looks fine
but routes queries to the wrong files. Better generation now = far less fine-tuning later.

**2. ALWAYS double-check the guard rules after generating.**
Even a good model sometimes skips the "competing intent guard" logic. After the profile
is generated, manually confirm these are actually present in the output file before you
trust it (this is the single most common gap):
- [ ] The COMPETING INTENT GUARD is applied across ALL 4 paths for every pair of
      subdomain intents that can fire on the same query — `pinned_owner_files`,
      `adjust_owner_score`, `adjust_document_score`, AND `pack_files_for_intents`.
      (A fix to pins alone is NOT enough — wrong files leak through the other 3 paths.)
- [ ] No domain words were dumped into `STOP_WORDS` in `search.py`
      (e.g. "phantom", "fatal", "locking", "ghost" must stay searchable).
- [ ] Every important subdomain has its OWN intent detector (not just top-level modules).
- [ ] Each alias phrase was checked so its injected tokens do not falsely fire a
      sibling intent in the same module.
If any of these are missing, the profile will surface wrong files on multi-symptom
queries — fix them by hand before going live, then re-test with your test runner.

---

## WHAT IS A PROFILE AND WHY DO YOU NEED ONE

ContextBridge (CB) is a code retrieval engine. Without a profile, it does pure keyword search — it finds files that contain your query terms, but ranks them by word frequency alone.

A profile teaches CB your domain: which files own which features, how to score them, and what context to give the analysis AI (Qwen/GPT/etc.). Without it, CB returns technically correct files but in the wrong order, and the analysis AI has no domain context.

**A profile is a one-time setup per project. Re-run this prompt only when you add a major new module.**

Read the project and identify:

- What are the main feature modules? List each one (e.g. Orders, Payments, Inventory, Users)
- For each module, what are the key backend service/controller files?

If Graphify output exists under graphify-out/, read the packs and behavior files there —
they describe the feature areas and invariants of this codebase.

---

STEP 2 — IDENTIFY THE HIGH-VALUE FILES PER MODULE

For each module you found in Step 1, list:
- The 2-3 most important backend files (controller + service)
- The 1-2 most important frontend files (main page or component)
- Any shared DTO/type files that are always relevant when that module is queried

For each module/subdomain, ALSO classify the files into these buckets:
- Primary mutation/state owner files
- Runtime consumer / job / resolver / policy files
- Bridge / handoff files that connect this module to another module
- Configuration editor / setup files
- Generic symptom / shell / overview / history files that are visible but usually not root cause

These become the "pinned files" — CB will always surface them when a query mentions that module.

IMPORTANT — pin the true owner files first:
- Prefer mutation/state/cache owner files over visible but secondary UI/configuration files.
- If a module has a state machine or lifecycle, pin the service/controller/API/client/entity files that mutate or validate that state before dashboard, report, print, or overview files.
- If Graphify source-files.txt lists controller/API, service, entity/model, DTO/type, cache/API client, and UI/worklist/page files, include the true mutation/state/cache owners first in pinned_owner_files.
- If a query pattern is about runtime behavior, stale config, lifecycle/session resolution, or downstream side effects, pin the runtime consumer / resolver / policy / job owners before configuration editor or admin setup files.
- If a bug requires tracing a handoff from one module to another, include the bridge/handoff files in the pinned set instead of only the upstream owner files.

---

STEP 3 — IDENTIFY INTENT SIGNALS (KEYWORDS)

For each module, list the words a developer would use when describing a bug in that module.
Examples:
- Orders module: "order", "cart", "checkout", "placed", "cancelled", "status"
- Payments module: "payment", "invoice", "refund", "charge", "settlement"
- Users module: "user", "login", "auth", "permission", "role", "access"

These become the intent detector tokens. CB uses them to decide which module a query is about.

IMPORTANT — also include operational/symptom tokens that a developer uses when filing a bug,
not just feature name tokens. Examples:
- Orders module: not just "order", "cart" — also "cache", "corrupt", "stuck", "blocked", "dispatch", "checkout"
- Payment module: not just "payment", "invoice" — also "failed", "timeout", "retry", "stuck", "pending"
- Shipping module: not just "shipping" — also "blocked", "override", "readiness", "eligibility"

Without these symptom tokens, a query like "order cart cache corruption" will NOT fire the
orders intent detector and CB will miss the order files entirely.

Also identify CROSS-MODULE CONFLICTS — pairs of modules where a symptom query might mention
both, but only one is the root cause. For example:
- "order cart + invoice print" → orders is root cause, invoice print is downstream symptom
- "payment failed + email not sent" → payment is root cause, email is downstream symptom

Also identify INTRA-MODULE SUBDOMAINS — within a single module, important sub-areas that
can each be an independent root cause and own their own files. Each important subdomain
needs its own intent detector. Example: within a large core module, "returns processing" and
"subscription coverage" are different subdomains — a subscription-coverage bug should not
route to returns-processing files, and vice versa. If a subdomain has no intent detector,
CB will miss its files entirely on subdomain-specific queries even if the parent module
intent fires.

For each important subdomain, explicitly identify:
- the primary owner files
- the runtime consumer files
- the bridge/handoff files
- the generic sibling files that must NOT outrank the true owner

CRITICAL SUBDOMAIN PATTERNS TO LOOK FOR:
- Runtime/configuration consumer vs configuration editor:
  when a query asks why a runtime behavior happened, prefer the runtime service/job/controller
  that consumes the config over the DTO/config editor file that merely stores it.
- Session/lifecycle resolver vs broad service:
  if a module has drawer/session/policy/runtime behavior, detect and pin the resolver/policy
  owner files, not only the broad controller/service.
- Billing/ledger/account-state vs generic billing pages:
  if the query mentions advance, customer account, payment applied, settlement, debit, or credit,
  treat that as a ledger/account-state subdomain with its own owner files.
- Package setup vs package consumption:
  if the query mentions session consumption, structured usage, service session, or package reduced,
  route to the runtime consumption path, not only the package setup/configuration files.
- Follow-up/notification runtime vs notification admin shell:
  if the query mentions follow-up outcome, reminder runtime, recipient scope, audit, scheduler, or
  automation config, pin the runtime services/jobs/controllers plus the config consumer file.
- Admin cashiering config/audit pages vs POS/core-module runtime neighbors:
  if the query asks about admin cashiering operations, audit handling, reports, permissions, or
  feature/runtime configuration, pin the admin cashiering page/API/permission owners before POS
  drawer UI or other core-module runtime neighbors.
- Module name may be omitted in real bug wording:
  if the workflow wording clearly identifies a subdomain even without the top-level module name,
  still create a dedicated intent. Examples: `event images` + `linked order details` should still
  route to the calendar product/media workflow; `service master definitions` + `staff availability`
  should still route to the service/staff owners; `report guard` + `exception filter` should still
  route to the module's report-guard/filter owners.
- Billing settlement / bill summaries vs generic billing history:
  if the query is about settlement, final payment, bill summaries, or summary updates, include the
  ledger controller/service and allocation engine owners in addition to billing owners.
- Backend/API phrasing vs UI/view phrasing:
  if the query explicitly asks for `API`, `backend components`, `controller`, or `service`,
  the subdomain intent must still fire and must pin/boost the backend owner files rather than
  only the visible UI pages for that area.
- Dashboard/global-search shell vs dashboard backend:
  if the query asks about dashboard loading, widget layout, shell, or global search entry,
  include the actual shell/layout/home files in addition to the dashboard backend/provider files.
- Legacy POS shell vs transactional POS runtime:
  if the query asks about shell, overlay, connection state, legacy compatibility, or sync UI,
  pin the shell/root POS UI files before checkout/sales/history files.
- POS drawer/runtime UI vs cashiering admin shell:
  if the query asks about POS drawer state, till runtime, cashiering context, or payment-side
  session behavior, pin the runtime POS UI owners (context panel, drawer modal, checkout modal)
  before admin cashiering tabs or generic movement pages.
- Workspace view/layout vs downstream billing/history pages:
  if the query asks about workspace loading, overview tabs, layout, context, or summaries,
  pin the workspace context/layout/util owners before history, billing, or print pages.
- Bootstrap/startup/runtime entry vs downstream services:
  if the query asks about terminal bootstrap, startup, initialization, runtime shell, or app entry,
  pin main/app/provider/bootstrap owners before cart/sales/runtime services.
- Product catalog sellable UI vs admin inventory maintenance:
  if the query asks about sellable inventory, product grid/display, POS product loading, or runtime stock display,
  pin the POS action/grid/card owners before admin inventory/catalog maintenance pages.
- Reports/audit UI vs report engines and adjacent domain-specific reports:
  if the query asks about reports, analytics, audit pages, filters, or admin report views,
  pin the actual report/audit UI owners before adjacent backend report engines or unrelated
  domain-specific report screens.
- Settings/runtime admin pages vs unrelated cashiering/runtime files:
  if the query asks about organization/store/terminal settings or runtime admin setup,
  pin the relevant settings pages and settings controllers before unrelated cashiering runtime files.
  Treat `terminal` and `runtime` as ambiguous words here: they often belong to cashiering/POS
  runtime intents too. Do not let `terminal + runtime` alone fire cashiering-style routing when
  the query also contains explicit admin-settings anchors like `organization`, `store`, `settings`,
  `configuration`, or `config`. Add a competing-intent guard so the admin settings intent wins.
- Resource-space / occupancy vs intake workflow:
  if the query asks about resource spaces, units, reservations, occupancy, or reserved-unit visibility,
  do not let the broad intake/booking workflow outrank the resource-space owner files. Add a competing
  guard so intake intents do not fire on resource-space reservation/occupancy wording.
- Core-module cashiering terminal vs admin cashiering runtime:
  if the query is about core-module billing/ledger actions blocked by terminal state, terminal context,
  or customer-side cashiering behavior, route to the core-module cashiering terminal owners and not the
  broad admin cashiering runtime owners.
- Runtime alone is not a module anchor:
  words like `runtime`, `config`, or `configuration` must not be enough by themselves to trigger
  POS/bootstrap/runtime shell intents. Require an explicit module or bootstrap anchor such as
  `pos`, `terminal`, `bootstrap`, `startup`, `main`, or `SettingsProvider` before routing there.
- Receipt template editor UI vs print-profile/nearby domain drift:
  if the query asks about receipt template editing, designer/list/editor pages, or print mapping,
  pin the receipt template UI owners before nearby print-profile backend files or unrelated
  party/customer files.
- Shell/view/history page vs true owner:
  if the query names a page, shell, workspace, overview, history, or audit screen, check whether the
  real root cause is actually a backend owner, runtime consumer, or bridge file that feeds that screen.
- Cross-module bridge / handoff:
  if module A triggers downstream effects in module B, create a dedicated bridge intent and identify
  the owner files on BOTH sides, especially the handoff file that connects them.
- Pack-injection completeness:
  for every subdomain intent you create, verify it is wired into `pack_files_for_intents` when a
  Graphify pack exists for that subdomain. A boost-only fix is incomplete if Graphify already has a
  dedicated pack and the profile never injects that pack's source files.

STEP 6 — GENERATE THE PROFILE FILE

Now write the complete profile file: rules/projects/<projectname>_profile.py

The file must implement these functions. Use the example_profile.py in this repo as a reference
for the exact function signatures and structure:

  pinned_owner_files(query_tokens: list[str]) -> list[str]
    - One intent detector per module (is_X_intent function)
    - Each intent detector checks for 2+ token sets (AND condition)
    - Returns a list of file paths to pin when that intent fires
    - Order matters: most important module first
    - For runtime/configuration/bridge intents, include the runtime consumer and bridge owners, not just the broad controller/service or config editor

  expand_query_tokens(query: str, tokens: list[str]) -> list[str]
    - Add domain-specific aliases to the search token list
    - Split compound tokens (e.g. "orderservice" → ["order", "service"])
    - Add synonyms: if "checkout" in tokens, also add "cart", "eligibility", "readiness"
    - Return the expanded token list

  module_intent_tokens() -> dict[str, set[str]]
    - Map module name → set of vocabulary tokens
    - Format: { "modulename": {"token1", "token2", ...} }
    - Include both feature names AND symptom/operational words per module
    - CRITICAL — THIS DRIVES THE GLOBAL DOMAIN GATE. The engine vote-counts query
      tokens against each module's vocabulary to decide which module a query is
      really about, then DROPS pinned files belonging to any other module. This is
      what stops a long multi-symptom query from pulling in unrelated modules'
      files. So each module's set must be RICH and DISCRIMINATIVE:
        * give every module enough distinctive tokens that a real query about it
          clearly out-votes modules merely mentioned in passing
        * avoid loading one module with generic words ("billing", "store", "user")
          that also belong to others — overlap dilutes the vote
        * a module with a thin/empty vocab can never win the gate, so its files
          will be filtered out even on its own queries. Cover EVERY module.

  adjust_document_score(score: float, normalized: str, query_tokens: list[str]) -> float
    - Broad doc-level scoring before owner scoring runs
    - Boost files strongly associated with the query domain
    - Suppress generic/shared files that frequently appear but are rarely root cause
    - Use path patterns (endswith, contains) to keep it fast
    - If you use intent detection here for module-specific boosts, apply the same
      competing-signal guard as in adjust_owner_score — an unguarded doc boost at
      this stage survives even after pin and owner score fixes, and wrong pack/module
      docs will still rank high
    - Code pattern for competing-signal guard in this function:
        token_set = set(query_tokens)
        _competing = is_X_intent(token_set) or is_Y_intent(token_set)
        if is_weaker_intent(token_set) and not _competing:
            if "weakmodule" in normalized or normalized.endswith("weakfile.cs"):
                score *= 3.0
                score += 500.0
      Never apply the doc boost unconditionally — always gate it with `not _competing`
      when a stronger subdomain intent can co-fire on the same query.

  adjust_owner_score(score, normalized, query_tokens, result, top_specific_packs) -> float
    - For each module intent that fires:
      - Multiply + add bonus for the PRIMARY file of that module (e.g. score *= 8.0; score += 4000)
      - Smaller bonus for secondary files
      - Suppress unrelated files (score *= 0.1) when the intent is specific
    - Cross-intent suppression: when module A is primary root cause,
      suppress module B files even if B is mentioned as a downstream symptom
    - COMPETING BOOST PATTERN: when two intents can fire together on the same query
      (e.g. a fulfillment bug mentions "checkout" so both checkout and fulfillment intents
      fire), use a comparative boost — full boost when the intent fires alone, reduced
      when a stronger competing intent is also active. Apply this to BOTH controller
      and service consistently:
        _competing = is_X_intent(token_set) or is_Y_intent(token_set)
        score *= 10.0 if not _competing else 1.5
        score += 6500.0 if not _competing else 150.0
      Never apply the full boost unconditionally when a competing subdomain intent exists.

  adjust_primary_owner_score(score, normalized, query_distinctive) -> float
    - Boost the most uniquely-named files for their module
    - Use endswith() on normalized file path
    - Target files whose name alone identifies them (e.g. "orderservice.cs", "checkoutpage.tsx")

  adjust_scoped_score(score: float, normalized: str, scoped_modules: set, scoped_packs: set) -> float
    - Stage-2 boost: runs AFTER the dominant module is identified from keyword results
    - If the file's module matches scoped_modules, apply a multiplier (e.g. score *= 1.3)
    - If the file's pack matches scoped_packs, apply a smaller multiplier
    - This pulls in supporting files from the winning module that keyword scoring may have missed

  extra_owner_file_patterns() -> tuple[str, ...]
    - Any API client or type files that should always be treated as high-priority owner files
    - (e.g. "apiClient.ts", "api.types.ts" — the main typed API client for the project)

  infer_module_from_path(path: str) -> str | None
    - Map a file's folder path to a module name
    - Used by the engine for hybrid fusion scoping AND the global domain gate
    - Format: if "/orders/" in path → return "orders"; if "/payments/" in path → return "payments"
    - Return None if the path does not match any known module
    - CRITICAL — THE DOMAIN GATE USES THIS TO FILTER PINS. After it picks the
      dominant module(s), it drops any pinned file whose inferred module is a
      DIFFERENT known module. A file that returns None is always kept (the gate
      only removes files it can prove are off-domain). Therefore:
        * cross-cutting files that live in generic folders (e.g.
          PermissionEvaluator.cs in /Services/, RoleController.cs in /Controllers/)
          will return None and slip through the gate as noise. Add explicit
          FILENAME-based fallbacks so these resolve to their true module:
            filename = path.rsplit("/", 1)[-1].lower()
            if any(n in filename for n in ("permissionevaluator","rolecontroller",...)):
                return "admin"
        * list every known noise file (auth/permission/role/user/print/report
          controllers and services) and bind it to the module it really belongs to,
          so the gate can suppress it on unrelated queries.

  low_signal_terms() -> tuple[str, ...]
    - Domain-specific words that dilute vector search when used as the full query
    - Examples: app name, module names, generic role names ("admin", "user", "manager")
    - These are filtered from the embedding query so the vector search focuses on meaning

  noise_files() -> dict
    - Files that rank high generically but rarely contain the root cause
    - Format: { "suffix": score_penalty } where suffix matches the end of the file path
    - Examples: app entry points, broad layout files, shared API clients, root controllers

  gap_queries() -> list[tuple[tuple[str, ...], str]]
    - Return the gap query table from Step 4
    - Format: [( ("trigger", "words"), "clean cb search query" ), ...]
    - Trigger words are matched against the query; if matched, CB uses the clean query instead

  pack_files_for_intents(query_tokens: list[str], graphify_root: str) -> list[str]
    - For each matched intent, read the corresponding Graphify pack's source-files.txt
    - Return those file paths so CB injects them even when keyword/vector scoring missed them
    - Use graphify_root to resolve pack paths: f"{graphify_root}/<module>/<pack>/raw/source-files.txt"
    - Example: f"{graphify_root}/orders/order-fulfillment-flow/raw/source-files.txt"
    - IMPORTANT: the layout is always <module>/<pack>/raw/source-files.txt — never packs/<packname>/source-files.txt
    - COMPETING GUARD: when a stronger competing intent fires, skip the weaker intent's
      pack injection using `continue` inside the loop — do NOT put None into the map
      (pack_rel is used as a path directly and will crash). Keep the map as static tuples.
      Evaluate the competing check at loop time:
        _competing = is_X_intent(token_set) or is_Y_intent(token_set)
        _guarded_packs = {"module/weak-intent-pack/raw/source-files.txt"}
        for intent_fn, pack_rel in intent_pack_map:
            if _competing and pack_rel in _guarded_packs:
                continue
            if intent_fn(token_set):
                ...

  analysis_prompt_override() -> str
    - Return the full system prompt from Step 5
    - Must be a complete replacement — include ALL sections from the generic prompt
      plus your domain-specific examples

IMPORTANT — REQUIRED OWNER COVERAGE PER INTENT
For every important intent/subdomain you generate, verify whether the intended result set includes:
- backend mutation/state owner
- runtime consumer / resolver / policy / job owner, if applicable
- bridge / handoff owner, if applicable
- UI/workflow owner only when it is genuinely required for debugging

If one of these owner layers is part of the real debugging path and you do not pin or boost it,
the profile is incomplete even if the module is technically correct.

- broad controller files
- shared helper files

If a query mentions a downstream symptom module, do not let that module win unless it owns the root cause. Example: if a state-machine/cache bug causes billing, print, calendar, notification, or reporting symptoms, the state/cache owner files must outrank those downstream symptom files.

IMPORTANT — AMBIGUOUS TERMS (CROSS-MODULE WORD CONFLICTS)
Some words exist in two or more modules with different meanings. These are the most dangerous source of ranking errors.

For every intent detector and alias phrase you write, ask:
  "Does this word appear in a DIFFERENT module with a different meaning?"

Examples of dangerous ambiguous words:
- "checkout" → could mean: POS sale transaction OR a guest/booking checkout completion
- "session" → could mean: user auth session OR a scheduled service session
- "order" → could mean: purchase order OR a service/work order
- "balance" → could mean: financial account balance OR workload balance
- "record" → could mean: a domain entity record OR database audit record

Rules for handling ambiguous terms:

1. INTENT DETECTORS must require a disambiguating second term — never fire on the ambiguous word alone.
   Bad:  is_pos_checkout_intent checks for "checkout" only
   Good: is_pos_checkout_intent requires "pos" OR "cart" OR "sale" in addition to "checkout"

2. Add an explicit EXCLUSION for the other module's context:
   def is_pos_checkout_intent(token_set):
       if token_set.intersection({"booking", "guest", "checkin", "checkout_stay", "reservation"}):
           return False  # booking checkout ≠ POS checkout
       ...

3. ALIAS PHRASES in expand_query_tokens that inject module-specific tokens must not fire
   in the other module's context. Add a domain context guard:
   _booking_context = bool(token_set.intersection({"booking", "guest", "checkin", "checkout_stay", "reservation"}))
   for required, additions in QUERY_ALIAS_PHRASES:
       if _booking_context and _POS_EXPANSION_TOKENS.intersection(additions):
           continue  # block POS token injection in booking queries

4. VERIFY: for each intent detector, write one test query from the OTHER module that contains
   the ambiguous word and confirm your intent detector returns False for it.

5. ALIAS-INJECTION CAN FIRE A SIBLING INTENT IN THE SAME MODULE — the domain gate
   only filters CROSS-module noise, so an intent in the SAME module triggered by an
   injected token is invisible to it and will surface as noise.
   Example bug: alias ("after","completion") injects "followup","reminder","notification".
   A sign/cache mutation query then has those tokens, so is_follow_up_intent fires and
   pins follow-up files — even though both are in the same core module.
   Rule: every intent detector must EXCLUDE the tokens that other intents inject when
   the real subject is different. Guard with the SUBJECT tokens, not the injected ones:
       def is_follow_up_intent(token_set):
           if token_set.intersection({"sign","signed","cache","corrupt","orderprocessing"}):
               return False  # a mutation/cache bug is not a reminder query
           ...
   For every alias phrase you add, list which intent its injected tokens could falsely
   trigger, and add that exclusion to the victim intent.

5b. SELF-FULFILLING ALIAS — an alias phrase designed to HELP an intent can make that very
   intent fire on queries genuinely about a different domain, if the alias injects the exact
   tokens the intent's detection condition requires.
   Example bug: alias ("product","catalog") injects "party","supplier","contact","service".
   is_party_catalog_dependency_intent requires (party|supplier) AND (product|catalog) AND service.
   Every query with "product"+"catalog" now satisfies all three conditions — including admin
   inventory queries — because the alias supplied the missing party and service tokens.
   The intent fires even though the query has no genuine party/catalog-dependency meaning.
   Rule: for every alias phrase, evaluate ALL intent detector functions against:
     (trigger_words + injected_tokens + plausible co-occurring query words from a DIFFERENT domain).
   If any intent fires that should NOT fire for that other-domain query:
     Option A — Remove the intent-triggering tokens from the alias. Aliases should inject
       SEARCH TOKENS (to improve keyword recall), not INTENT TOKENS (which fire routing logic).
       A lean alias ("product","catalog") → ("service","inventory") lets the intent fire only
       when genuine party/supplier context appears in the actual query, not from injected tokens.
     Option B — Add a domain exclusion guard to the intent using tokens the alias does NOT inject
       (so the alias cannot satisfy the exclusion — a token that only appears when the query is
       genuinely about that other domain, e.g. "admin" for admin-product queries):
         def is_party_catalog_dependency_intent(token_set: set[str]) -> bool:
             if bool(token_set.intersection({"admin", "inventory", "uom", "pack", "batch"})):
                 return False  # admin product-catalog queries must not trigger party pins
             ...
   Always prefer Option A when alias additions can be simplified to search-only tokens.
   Use Option B when the alias must retain the intent tokens for other valid sub-queries and
   the exclusion signal is highly reliable (present in ALL wrong-domain queries).

6. GENERIC NOUNS MUST NOT BE SOLE DISAMBIGUATORS in any AND condition — words like "audit",
   "catalog", "product", "admin", "record" appear in file names across unrelated modules and
   will match any query containing that word, even if the query is about a completely different
   domain. Never put a generic noun as the ONLY token in one half of an AND intent condition.
   Bad:  is_admin_access_intent second condition includes "catalog" — so "admin product catalog
         pack-sizing uom" fires the permission intent (PermissionCatalog/PermissionEvaluator get
         pinned) instead of the inventory intent. "catalog" alone qualified because it also names
         the PermissionCatalog.cs file.
   Good: Remove the generic noun from the second condition. Specific tokens like "evaluator",
         "login", "auth" are sufficient to detect the permission domain without "catalog". If
         "catalog" can also mean product catalog, it needs its own dedicated intent (rule 7).

7. EVERY NOUN-DOMAIN GETS ITS OWN INTENT — if a noun like "catalog" can mean:
   (a) PermissionCatalog (access control), (b) product/inventory catalog, (c) dashboard widget
   catalog — each meaning needs a dedicated intent detector with domain-specific co-tokens.
   Never add a generic noun to an existing intent's AND conditions hoping it will self-limit.
   Before adding a noun to any intent condition, ask: "Could this noun appear in a query that
   is clearly about a different domain?" If yes, create a new intent for that domain and add a
   competing guard between them (see rule below). Then remove the noun from the original intent
   and verify the original intent still fires correctly on its own domain queries.

WHY THIS MATTERS — pins are scored in strict order bands (earlier pin = higher band,
never overtaken). If a wrong intent fires, its files get pinned and routed deterministically.
The domain gate removes wrong-MODULE pins; wrong-intent-SAME-module pins must be blocked at
the intent detector (rules 2 and 5) or ordered after the true owner in pinned_owner_files.

IMPORTANT — COMPETING INTENT GUARD — ALL 4 PATHS
When two intents within the same module can fire simultaneously on the same query
(e.g. a lifecycle/workflow term like "completed" or "blocked" appears in a bug report
about a different subdomain like fulfillment or ledger), the weaker intent's files must
be suppressed across ALL FOUR paths — not just pins:

  Path 1 — pinned_owner_files():
    Add exclusion guard: if is_weaker_intent and not (is_stronger_intent_1 or is_stronger_intent_2)

  Path 2 — adjust_owner_score():
    Use comparative boost: full multiplier when intent fires alone, small multiplier when competing

  Path 3 — adjust_document_score():
    Add the same competing-signal guard to any intent-based doc boost in this function

  Path 4 — pack_files_for_intents():
    Skip the weaker intent's pack using `continue` when competing intents are active

Missing any one of these 4 paths leaves a back-door for wrong files to surface.
A fix to pins alone is not sufficient — document score and pack injection must be guarded too.

8. KEYWORD-NOISE SUPPRESSION IN adjust_document_score — generic words that exist in FILE NAMES
   across unrelated modules will cause those files to score high on any query containing that
   word, even when the intent is completely different.
   Example: "audit" exists in NotificationAuditLog.cs and NotificationAuditPage.tsx. An unrelated
   query like "why does approvedAt serve as the audit truth for the reviewer draft" scores those
   notification files high via keyword match on "audit" — even though the query has nothing to
   do with notification plumbing.
   Rule: for every generic word that appears in file names across module boundaries, add a
   suppression rule in adjust_document_score that fires when the word is in the query but no
   domain-context tokens for that module are present.

   CRITICAL — USE AN EXTENDED SCAN VARIABLE, NOT THE SHARED `lowered` VARIABLE:
   The standard `lowered` variable built at the top of adjust_document_score only covers the
   first 500 characters of doc_text. Graphify source-files.txt documents list pack files one
   per line — the file name you want to suppress may appear at line 20+ (well beyond 500 chars).
   Always build a SEPARATE extended-scan string for noise suppression and use ≤0.01× (NOT 0.1×):
     _noise_scan = f"{(doc_path or '').lower()} {(doc_pack or '').lower()} {(doc_text or '').lower()[:3000]}"
     if "<generic_word>" in token_set and not token_set.intersection(<domain_context_tokens>):
         if any(noise in _noise_scan for noise in (<cross-module file names containing the word>)):
             score *= 0.01   # ≤0.01x — 0.1x is too weak; high-frequency noise tokens still outrank good files
             reasons.append("suppressed <module> noise for non-<module> <word> query")
   Do NOT reuse `lowered` for this — it is intentionally capped at 500 chars and will miss
   file references that appear mid-document in source-files.txt packs.
   Domain-context tokens for a module are the tokens that ONLY appear when someone is genuinely
   asking about that module (e.g. for notification module: "notification", "email", "sms",
   "reminder", "followup", "recipient"). The generic word alone ("audit") is NOT a domain token.

8b. CROSS-DOMAIN TOKEN COLLISION IN INTENT AND CONDITIONS — before adding any token to an
   intent's AND condition set, search ALL file names in the project for that token. If the
   token appears in file names from a DIFFERENT domain, it is a cross-domain noun. It MUST
   NOT be in any existing intent's condition set — it needs its own dedicated intent.
   Example: "catalog" appears in AccessCatalog.cs (access-control), DashboardWidgetCatalogResolver.cs
   (admin), product-catalog packs (inventory), and can match party-catalog docs. Adding "catalog"
   to a party-contact intent's second AND condition caused it to fire on product-catalog queries,
   ranking VendorService.cs and SupplierController.cs at positions 1-2 for inventory queries.
   Fix: remove the cross-domain noun from the existing intent and verify the intent still fires
   on its own domain queries using only the remaining tokens. Then add a product-catalog intent
   with "catalog" as one token AND inventory-specific co-tokens (e.g. "product", "inventory",
   "uom", "batch", "sku") so it cannot fire without genuine product context.
   Rule: every intent detector must be safe to evaluate against a query from the WRONG module
   that contains the shared noun. Write that test query and confirm the detector returns False.

9. CROSS-MODULE BRIDGE INTENTS — when a core-module action triggers automatic
   effects in a secondary module (e.g. order confirmation → cashiering auto-collection,
   POS checkout → calendar handoff), the primary module dominates ranking and secondary module
   files get suppressed. Add a dedicated bridge intent:
     def is_A_B_bridge_intent(token_set: set[str]) -> bool:
         # Requires at least one token from each module — no false-fires on single-module queries.
         _primary_tokens = {"order", "orderconfirm", "sign", "signed", "charge", "charging"}
         _secondary_tokens = {"cashiering", "autocollect", "autocollection", "collection"}
         return bool(token_set.intersection(_secondary_tokens)) and bool(token_set.intersection(_primary_tokens))
   Wire the bridge intent into:
   - pinned_owner_files(): pin the secondary module's key files (CashieringController.cs,
     CashieringService.cs) — this guarantees they surface even if the domain gate drops them
   - adjust_owner_score(): boost the secondary module's files without suppressing primary files
   Do NOT add a suppression rule for the primary module inside this intent — the bridge must
   lift the secondary without displacing the primary. Use moderate boost multipliers (5x + ~2000pts)
   rather than the highest-tier boosts reserved for single-module intents.

IMPORTANT — STOP_WORDS AND DOMAIN VOCABULARY
Never add domain-specific vocabulary to STOP_WORDS in search.py. STOP_WORDS is for
generic English words only ("the", "and", "for", etc.). Domain terms like "phantom",
"fatal", "locking", "ghost", "balance", "record" may look like noise but are often
trigger tokens for intent detectors. Stripping them silently breaks intent detection
before any profile logic runs. If a term is noisy in context, handle it with suppression
rules in the profile — not by removing it from the token stream globally.
